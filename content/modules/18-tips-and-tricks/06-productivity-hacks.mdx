## The Hacks Nobody Reads the Docs For

I have been using Claude Code daily for months. These are the tricks I wish someone had told me on day one. Each one saves a small amount of time, but compounded across hundreds of daily interactions, they transform your workflow.

## Hack 1: /init for Instant Project Context

Starting a new project or joining an existing one? Do not spend 20 minutes writing a CLAUDE.md from scratch.

```bash
claude
> /init
```

Claude scans your project — `package.json`, directory structure, config files, existing docs — and generates a tailored CLAUDE.md. In 30 seconds you get what would take 20 minutes to write manually.

Then edit the generated file to add anything project-specific that Claude could not infer.

## Hack 2: Screenshot Paste for UI Bugs

Stop describing visual bugs in text. Paste a screenshot.

```bash
claude
> "This button should be aligned to the right side of the card, 
   but look at what's happening:"
> [paste screenshot]
> "Fix the alignment in @src/components/Card.tsx"
```

Claude sees the screenshot, understands the layout problem, and fixes the CSS. This works for:
- Layout bugs
- Responsive design issues
- Color/typography mismatches
- Component spacing problems
- Visual regressions

## Hack 3: /review Before Every PR

Make it a habit — before creating a PR, run:

```bash
/review
```

Claude reviews your staged changes and catches:
- Bugs you missed
- Missing error handling
- Security issues
- Leftover debug code (console.logs, TODOs)
- Inconsistencies with project patterns

I caught a SQL injection vulnerability this way that would have been embarrassing in a team code review.

## Hack 4: Shell Aliases for Common Workflows

Add these to your `.zshrc` or `.bashrc`:

```bash
# Quick fix mode
alias cfix='claude -p "Fix the failing tests in this project. Run the tests, identify failures, and fix them."'

# Code review
alias creview='claude -p "Review the git diff for issues: bugs, security, performance. Be concise."'

# Quick documentation
alias cdoc='claude -p "Add JSDoc comments to all exported functions in"'

# Explain a file
alias cexplain='claude -p "Explain what this file does, its key functions, and how it fits in the project:"'

# Generate tests
alias ctest='claude -p "Generate comprehensive unit tests for"'
```

Usage:

```bash
cfix                           # Fix all failing tests
cdoc src/lib/auth.ts          # Document a file
ctest src/lib/utils.ts        # Generate tests
cexplain src/middleware.ts    # Understand a file
```

## Hack 5: Pipe Everything

Claude Code accepts piped input. This opens up powerful compositions:

```bash
# Analyze test failures
npm test 2>&1 | claude -p "Analyze these test failures and suggest fixes"

# Explain a git diff
git diff | claude -p "Summarize these changes for a PR description"

# Analyze logs
tail -100 /var/log/app.log | claude -p "Find error patterns in these logs"

# Review a specific commit
git show HEAD | claude -p "Review this commit for issues"

# Explain a complex command
history | tail -20 | claude -p "Explain what each of these commands does"
```

## Hack 6: The CLAUDE.md Bookmark System

Save your best prompts in CLAUDE.md so you never have to re-craft them:

```markdown
# CLAUDE.md

## My Prompt Templates

### Code Review Checklist
When I say "review this", check for:
1. Null/undefined handling
2. Error boundaries
3. Input validation
4. SQL injection / XSS
5. Missing loading/error states
6. Console.logs left in
7. Hardcoded values that should be env vars

### Test Generation Rules
When I say "test this", generate:
- Happy path tests
- Error cases (invalid input, network failure, timeout)
- Edge cases (empty arrays, null values, max values)
- Use describe/it blocks with clear names
- Mock external dependencies
```

Now every time you say "review this" or "test this," Claude follows your personal checklist automatically.

## Hack 7: Multi-Cursor Editing + Claude

Use Claude to generate repetitive code, then multi-cursor to adapt it:

```bash
claude -p "Generate 10 similar API route handlers for these entities:
users, projects, tasks, comments, tags, categories, 
attachments, notifications, settings, audit-logs

Each should follow the pattern in @src/app/api/users/route.ts"
```

Then use your editor's multi-cursor to make entity-specific tweaks. Faster than writing each one manually or having Claude handle all the edge cases.

## Hack 8: The "Show Me Don't Tell Me" Command

```bash
# Instead of asking Claude to explain:
"Show me an example of using the Supabase client with RLS"

# Be specific about what "show" means:
"Write a working example file at src/examples/rls-demo.ts that 
demonstrates: creating a row with RLS, querying with RLS, 
and what happens when RLS blocks access. Include the SQL 
policy definitions as comments."
```

Working examples teach faster than explanations.


## Hack 10: The "Fix It For Me" Pattern

When you have an error, do not ask Claude to explain — ask it to fix:

```bash
# Instead of:
"Why am I getting 'TypeError: Cannot read property of undefined'?"

# Do this:
claude -p "Fix this error in @src/components/Dashboard.tsx:
TypeError: Cannot read properties of undefined (reading 'map')
at Dashboard (Dashboard.tsx:42:18)

Add proper null checks and loading states."
```

Direct fix requests are faster and use fewer tokens than explain-then-fix.

## Hack 11: Use --output for File Generation

```bash
# Generate a file directly
claude -p "Generate a comprehensive .gitignore for a 
Next.js + Supabase project" --output .gitignore

# Generate config files
claude -p "Generate a Playwright config for testing 
a Next.js app on localhost:3000" --output playwright.config.ts
```

## Hack 12: Context Windows as Scratchpads

Use a Claude session as a temporary workspace for complex thinking:

```bash
claude
> "I need to design the data flow for our checkout process.
   Let me think out loud and you help me refine:
   
   Cart → Validate Stock → Calculate Total → Apply Discount → 
   Create Order → Process Payment → Send Confirmation
   
   What am I missing?"

# Claude identifies gaps: tax calculation, inventory reservation,
# payment failure handling, idempotency keys
```

This is faster than writing it out in a doc because Claude actively fills in gaps.

## Hack 13: The Pre-Flight Check

Before deploying or merging, run a quick sanity check:

```bash
claude -p "Pre-flight check for @src/app/api/:
1. Are there any console.logs that should be removed?
2. Are there any TODO comments that should be addressed?
3. Are there any hardcoded secrets or API keys?
4. Are there any functions without error handling?
5. Are there any unused imports?
List only actual issues found."
```

## Hack 14: Learning New Codebases Fast

When joining a project or reviewing unfamiliar code:

```bash
claude
> "I just joined this project. Give me a 2-minute overview:
   1. What does this project do?
   2. What's the tech stack?
   3. What's the file structure pattern?
   4. Where are the main entry points?
   5. What are the key abstractions I should understand?"
```

Claude reads your entire project structure and gives you a guided tour.

## Hack 15: The Rubber Duck with Superpowers

```bash
claude
> "I'm stuck on how to handle concurrent writes to the 
   same document. Here's my current approach: [describe].
   
   Poke holes in it. What will break? What am I not thinking about?"
```

Claude is a rubber duck that talks back with genuinely useful criticism.

## Hack 16: Workspace-Specific CLAUDE.md

Keep different CLAUDE.md files for different contexts:

```
~/work/frontend/CLAUDE.md    → React patterns, component library
~/work/backend/CLAUDE.md     → API conventions, database patterns
~/personal/blog/CLAUDE.md    → Content style, MDX format
~/freelance/client/CLAUDE.md → Client-specific requirements
```

Each workspace gets its own context automatically.

## Hack 17: Quick Prototyping

```bash
claude -p "Create a minimal working prototype of a real-time 
chat app using Next.js and Supabase Realtime.

Files needed:
1. src/app/chat/page.tsx — chat UI
2. src/hooks/useChat.ts — real-time message hook
3. supabase/migrations/chat.sql — messages table

Make it actually work, not just scaffolding."
```

From idea to working prototype in one prompt.

## Hack 18: The "Diff Only" Output

When you want to see exactly what Claude would change:

```bash
claude -p "Show me a diff (unified format) of the changes needed 
to add input validation to @src/app/api/users/route.ts. 
Don't modify the file — just show the diff."
```

Review the diff first, then decide whether to apply it.

## Hack 19: Session Checkpoints

Before risky operations, create a mental checkpoint:

```bash
# Before a big refactor
git stash  # or commit WIP
claude "Refactor the entire auth module..."

# If it goes wrong
git stash pop  # or reset to your commit
# Start fresh with a better prompt
```

Always have a way to go back.

## Hack 20: The Productivity Stack

My daily Claude workflow, in order:

```
Morning:
1. /init (if new project) or review CLAUDE.md
2. Check failing tests: cfix alias
3. Plan the day's work: /plan "Today I need to..."

During development:
4. Headless mode for known tasks
5. Interactive mode for exploration
6. /compact every 10-15 messages
7. Screenshot paste for UI work

Before committing:
8. /review to catch issues
9. Generate commit message from diff
10. /simplify for final cleanup

End of day:
11. Generate PR descriptions
12. Update CLAUDE.md with new learnings
```

## Key Takeaway

Productivity with Claude is not about one big trick — it is about twenty small ones that compound. Set up your aliases, learn the CLI flags, build your CLAUDE.md into a personal knowledge base, and develop mode-switching habits. The developers who are 10x more productive with Claude are not writing better prompts — they have built systems around Claude that eliminate friction from their entire workflow. Start with hacks 1, 4, and 6 today, and add one new hack each week.
