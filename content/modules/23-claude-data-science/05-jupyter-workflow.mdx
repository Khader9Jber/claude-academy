## Two Tools, One Workflow

Jupyter notebooks are where data science happens — interactive, visual, iterative. Claude Code is where clean code gets written — structured, testable, production-ready. The magic is using both together.

## The Side-by-Side Workflow

Here's the setup I use daily: Jupyter open in a browser tab, Claude Code open in a terminal beside it.

```
+---------------------------+---------------------------+
|     Jupyter Notebook      |      Claude Code          |
|                           |                           |
|  [In 1]: df.head()        |  > claude "Generate a     |
|  [Out 1]: ...             |    function that cleans    |
|                           |    this date column"       |
|  [In 2]: # paste from     |                           |
|          # Claude          |  [Returns clean function] |
|                           |                           |
|  [In 3]: clean_df.plot()  |  > claude "Why is this    |
|  [Out 3]: [chart]         |    merge dropping rows?"  |
+---------------------------+---------------------------+
```

The workflow:
1. Explore in Jupyter — run quick commands, view data, plot things
2. Hit a wall — complex transformation, weird error, need a function
3. Ask Claude Code — describe what you need, paste in the error
4. Copy result back to Jupyter — paste the generated code, run it
5. Repeat

It sounds simple, but this loop runs 10x faster than Googling + StackOverflow.

## Notebook to Production Script

This is one of the highest-value transformations Claude does. Every data scientist has notebooks that "work" but can't be scheduled, tested, or integrated.

```bash
claude "Convert ./notebooks/daily_report.ipynb to a production
Python script. It should:

1. Be a proper Python module with functions, not a flat script
2. Accept command-line arguments for the date range and output path
3. Add logging instead of print statements
4. Add error handling for missing data and API failures
5. Include a main() function and __name__ guard
6. Add type hints
7. Generate the same PDF report the notebook produces"
```

Claude reads the notebook, understands the flow, and restructures it:

```python
"""Daily sales report generator.

Converted from daily_report.ipynb.
Usage: python daily_report.py --start 2024-01-01 --end 2024-01-31
"""
import argparse
import logging
from datetime import datetime
from pathlib import Path
from typing import Optional

import pandas as pd
import matplotlib.pyplot as plt
from matplotlib.backends.backend_pdf import PdfPages

logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s [%(levelname)s] %(message)s'
)
logger = logging.getLogger(__name__)


def load_data(start_date: str, end_date: str) -> pd.DataFrame:
    """Load sales data for the given date range."""
    try:
        df = pd.read_parquet("./data/sales.parquet")
        df['date'] = pd.to_datetime(df['date'])
        mask = (df['date'] >= start_date) & (df['date'] <= end_date)
        filtered = df[mask]
        logger.info(f"Loaded {len(filtered)} rows for "
                    f"{start_date} to {end_date}")
        return filtered
    except FileNotFoundError:
        logger.error("Sales data file not found")
        raise


def compute_metrics(df: pd.DataFrame) -> dict:
    """Calculate key business metrics."""
    return {
        'total_revenue': df['amount'].sum(),
        'order_count': len(df),
        'avg_order_value': df['amount'].mean(),
        'unique_customers': df['customer_id'].nunique(),
    }


def generate_charts(df: pd.DataFrame,
                    pdf: PdfPages) -> None:
    """Generate report charts and add to PDF."""
    # Revenue trend
    fig, ax = plt.subplots(figsize=(12, 6))
    daily = df.groupby('date')['amount'].sum()
    daily.plot(ax=ax, color='#2563EB', linewidth=2)
    ax.set_title('Daily Revenue')
    ax.set_ylabel('Revenue ($)')
    pdf.savefig(fig)
    plt.close(fig)

    # Category breakdown
    fig, ax = plt.subplots(figsize=(10, 6))
    df.groupby('category')['amount'].sum().sort_values().plot(
        kind='barh', ax=ax, color='#3B82F6')
    ax.set_title('Revenue by Category')
    pdf.savefig(fig)
    plt.close(fig)

    logger.info("Generated report charts")


def main(start_date: str, end_date: str,
         output_path: Optional[str] = None) -> None:
    """Generate the daily sales report."""
    if output_path is None:
        output_path = f"./reports/sales_{start_date}_{end_date}.pdf"

    Path(output_path).parent.mkdir(parents=True, exist_ok=True)

    logger.info(f"Generating report: {start_date} to {end_date}")

    df = load_data(start_date, end_date)
    if df.empty:
        logger.warning("No data for the given date range")
        return

    metrics = compute_metrics(df)
    logger.info(f"Revenue: ${metrics['total_revenue']:,.2f} | "
                f"Orders: {metrics['order_count']}")

    with PdfPages(output_path) as pdf:
        generate_charts(df, pdf)

    logger.info(f"Report saved to {output_path}")


if __name__ == "__main__":
    parser = argparse.ArgumentParser(
        description="Generate daily sales report")
    parser.add_argument("--start", required=True,
                        help="Start date (YYYY-MM-DD)")
    parser.add_argument("--end", required=True,
                        help="End date (YYYY-MM-DD)")
    parser.add_argument("--output", help="Output PDF path")
    args = parser.parse_args()
    main(args.start, args.end, args.output)
```

Same logic, now schedulable with cron, testable with pytest, integratable with Airflow.

## Debugging Data Issues

The most valuable debugging skill: figuring out why your numbers changed.

```bash
claude "My model accuracy dropped from 0.92 to 0.71 after
retraining on this month's data. Same code, new data.

Generate a diagnostic script that:
1. Loads old training data (./data/train_v1.csv)
   and new training data (./data/train_v2.csv)
2. Compares distributions of each feature (KS test)
3. Checks if the target variable distribution changed
4. Identifies new categorical values not seen in v1
5. Compares missing value patterns
6. Produces a drift report"
```

Claude generates a comprehensive drift detection script:

```python
from scipy import stats

def detect_drift(old_df, new_df, numeric_cols, cat_cols,
                 target_col):
    """Compare two datasets and report drift."""
    report = []

    # Target distribution
    old_target = old_df[target_col].mean()
    new_target = new_df[target_col].mean()
    target_shift = abs(new_target - old_target)
    report.append(f"Target rate: {old_target:.3f} → "
                  f"{new_target:.3f} (shift: {target_shift:.3f})")
    if target_shift > 0.05:
        report.append("  ⚠ SIGNIFICANT target drift detected!")

    # Numeric feature drift (KS test)
    report.append("\nNumeric Feature Drift:")
    for col in numeric_cols:
        stat, pval = stats.ks_2samp(
            old_df[col].dropna(), new_df[col].dropna())
        drift = "DRIFT" if pval < 0.01 else "OK"
        report.append(f"  {col}: KS={stat:.3f}, "
                      f"p={pval:.4f} [{drift}]")

    # Categorical drift
    report.append("\nCategorical Feature Drift:")
    for col in cat_cols:
        old_vals = set(old_df[col].dropna().unique())
        new_vals = set(new_df[col].dropna().unique())
        new_categories = new_vals - old_vals
        if new_categories:
            report.append(f"  {col}: NEW values: {new_categories}")

    # Missing value pattern changes
    report.append("\nMissing Value Changes:")
    for col in old_df.columns:
        old_missing = old_df[col].isna().mean()
        new_missing = new_df[col].isna().mean()
        if abs(new_missing - old_missing) > 0.05:
            report.append(f"  {col}: {old_missing:.1%} → "
                          f"{new_missing:.1%}")

    return "\n".join(report)
```

You run it, find out that the target rate shifted from 20% to 35% because of a seasonal effect, and three features have new categories the model never saw. Mystery solved in 5 minutes instead of 5 hours.

## The Golden Rule

Notebooks are for thinking. Scripts are for doing. Claude helps you think faster in notebooks and converts your thinking into scripts that do. Don't try to make notebooks production-ready. Don't try to explore data in scripts. Use each tool for what it's good at, and let Claude bridge the gap.

## Key Takeaway

The Claude Code + Jupyter workflow keeps each tool in its sweet spot: Jupyter for interactive exploration and visualization, Claude Code for code generation, debugging, and refactoring. Claude reads `.ipynb` files directly, giving it full context of your analysis including outputs and errors. The highest-value use case is converting working notebooks into production scripts with proper structure, error handling, logging, and CLI arguments. When debugging data issues like model accuracy drops, Claude generates drift detection scripts that compare old and new data distributions — because model degradation is almost always a data problem, not a model problem.
