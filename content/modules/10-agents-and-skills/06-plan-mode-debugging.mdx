## Plan Before You Build

The most common mistake developers make with Claude Code is jumping straight to implementation. "Build me a payment service" produces mediocre results. "Let's plan the payment service first, then build it" produces excellent results.

Plan mode gives you a structured way to think before you code. In Claude 4.6 it's also the first stop in the **4-phase workflow**: Explore → Plan → Implement → Commit.

## Entering Plan Mode

Enter plan mode one of two ways:

```bash
# Start a session already in plan mode
claude --permission-mode plan

# Or press Shift+Tab during a session to cycle modes until you land on plan
```

In plan mode, Claude:
- **Can**: Read files, analyze code, search the codebase, run read-only commands
- **Cannot**: Write files, modify code, run destructive commands

This is deliberately limiting. The constraint forces Claude to think, not act.

Need to revise the plan yourself? Press `Ctrl+G` to open it in your editor. Edit, save, and the updated plan comes back into the session.

### Using Plan Mode

```bash
claude --permission-mode plan

"I need to add order cancellation with refund processing to our API.
Read the existing order service, payment service, and database schema.
Then create a detailed plan covering:
1. Database changes needed
2. New service methods
3. API endpoint design
4. Integration with existing payment processing
5. Test strategy
6. Migration plan"
```

Claude reads the codebase, understands the existing architecture, and produces a thorough plan — without touching any files.

### Reviewing the Plan

Claude's plan might look like:

```markdown
## Order Cancellation Plan

### Database Changes
- Add `canceled_at` and `cancellation_reason` columns to orders table
- Add `refund_id` column to link cancellation to refund
- New migration: 20260404_add_order_cancellation.sql

### Service Layer
- OrderService.cancelOrder(orderId, reason): 
  - Validates order is in cancellable state
  - Creates refund via PaymentService
  - Updates order status to 'canceled'
  - Emits OrderCanceled event

### API Endpoint
- POST /api/v1/orders/:id/cancel
- Body: { reason: string }
- Response: { order: Order, refund: Refund }
- Auth: requires order owner or admin

### Test Strategy
- Unit: cancelOrder service method (happy path, already canceled, 
  non-cancellable status, refund failure)
- Integration: full endpoint test with database
- Edge: concurrent cancellation attempts
```

You can review, ask questions, and refine the plan before any code is written.

### Executing the Plan

When you're satisfied with the plan, press Shift+Tab to leave plan mode (or tell Claude "go ahead and implement this"). Claude switches to full tool access and implements the plan. The plan stays in context, so it follows the architecture, naming, and patterns you agreed on.

This is the **4-phase workflow** in action:

1. **Explore** — Claude reads the relevant code (still in plan mode).
2. **Plan** — Claude proposes the approach; you review and edit via `Ctrl+G`.
3. **Implement** — Leave plan mode; Claude writes code against the plan.
4. **Commit** — `/commit` or manual git, with the plan as the context for a good message.

## Parallel Debugging with /rewind and Worktrees

When you have multiple hypotheses about a bug, you have two tools for exploring them in parallel.

**`/rewind`** restores a conversation or code checkpoint — and it persists across sessions. Try hypothesis A; if it's wrong, `/rewind` to the pre-hypothesis state and try hypothesis B. You pick what to rewind: conversation, code, or both.

```bash
# Baseline
"I think the timeout is in the database query. Let's check order-repo.ts..."

# Dead end. Rewind conversation + code.
/rewind

# Try the other theory
"Actually, let's check the Redis connection instead..."
```

For hypotheses that modify files in conflicting ways, use **git worktrees** so each experiment runs in its own directory:

```bash
git worktree add ../theory-db feature/debug-db
git worktree add ../theory-redis feature/debug-redis
# Start a Claude session in each; they can't contaminate each other
```

## The Systematic Debugging Approach

When debugging with Claude, follow this structure:

### Step 1: Reproduce

```bash
"The bug: users report seeing 'null' for their order total. 
Here's the error log: @logs/api-errors-2026-04-04.log
Can you reproduce this by finding the code path?"
```

Claude traces the code path to understand how the bug manifests.

### Step 2: Isolate

```bash
"Narrow it down. Is the null coming from:
a) The database (stored as null)
b) The service layer (calculation error)
c) The API serialization (formatting issue)"
```

Claude reads each layer and isolates where the null originates.

### Step 3: Diagnose

```bash
"We've isolated it to the service layer. The calculateTotal function
returns null when the order has no items. Read the function and explain
why — what's the root cause?"
```

Claude analyzes the specific function and identifies the root cause.

### Step 4: Fix

```bash
"Fix it. The calculateTotal function should return 0 for empty orders, 
not null. Also add a test that catches this case."
```

Claude implements the fix and writes the test.

### Step 5: Verify

```bash
"Run the tests. Also check if there are other places in the codebase 
where we make the same mistake — returning null instead of a default 
value for calculations."
```

Claude runs tests and searches for similar issues.

## Combining Strategies

The most effective debugging sessions combine multiple strategies:

```bash
# Start in plan mode to understand the problem
claude --permission-mode plan
"Analyze the payment processing code and identify all places where
a timeout could occur. Don't fix anything yet — just map the risks."

# Review the analysis. Edit the plan with Ctrl+G if needed.

# Leave plan mode (Shift+Tab) and implement
"Fix the timeout in the webhook handler — add a 30-second timeout
with retry logic"

# If the fix is risky, set a checkpoint before running it
"Run the payment test suite"

# If tests fail, /rewind to the pre-fix state and try a different approach
```

Explore → Plan → Implement → Verify, with `/rewind` as the safety net. Each tool serves a specific purpose.

## Key Takeaway

Plan mode (entered via `--permission-mode plan` or `Shift+Tab`) gives Claude read-only access for analysis and planning before making changes. `Ctrl+G` opens the plan in your editor for manual revision. Leave plan mode to implement. Use `/rewind` (persistent across sessions) to branch debugging hypotheses, and git worktrees when hypotheses conflict at the file level. The systematic debugging approach — reproduce, isolate, diagnose, fix, verify — plus the 4-phase workflow gives Claude the structure it needs to find root causes efficiently.
