## The Onboarding Problem

The typical new developer experience: clone the repo, read a stale README, spend two days getting the dev environment working, spend a week reading code, and start being productive in week three. With Claude Code, you can compress this to two days.

Here is the onboarding workflow I have used across three teams.

## The Onboarding Kit

Every team using Claude Code should prepare these four things for new developers:

### 1. A CLAUDE.md With an Onboarding Section

Add this to the bottom of your project CLAUDE.md:

```markdown
## Onboarding Guide (New Developers)

### First Hour: Setup
1. Clone the repo and run `pnpm install`
2. Copy `.env.example` to `.env.local` and fill in values (ask #team-backend for secrets)
3. Run `npx prisma migrate dev` to set up the local database
4. Run `pnpm dev` and verify http://localhost:3000/health returns 200
5. Run `pnpm test` and verify all tests pass

### First Day: Explore
Ask Claude these questions to understand the codebase:
- "Explain the project architecture. What does each directory in src/ do?"
- "Trace a request through POST /api/v1/orders from the route handler to the database write. Show me every file involved."
- "What is the authentication flow? How does a user go from login to making authenticated API calls?"
- "Show me the error handling pattern. What happens when a service throws a NotFoundError?"
- "What testing patterns does this project use? Show me a typical test file."

### First Week: Contribute
Your first task should be small and touch every layer:
- Add a new field to an existing model
- Add a new endpoint to an existing route file
- Write tests for an untested function
- Fix a bug from the backlog labeled "good first issue"

### Common Gotchas
- The test database resets between runs — don't store test data you need
- Prisma migrations must be backward-compatible (see docs/database.md)
- Always create a feature branch — main is protected
- The CI pipeline takes ~4 minutes — don't wait for it, start the next task
```

### 2. Starter Prompt Library

Create a file at `.claude/prompts/onboarding.md` with ready-made prompts:

```markdown
# Onboarding Prompts

## Architecture Exploration
"Give me a high-level overview of this project's architecture. 
What are the main components, how do they interact, and what 
technology does each use?"

## Trace a Feature
"Trace the complete flow when a user creates a new order:
1. Which route handles the request?
2. What validation happens?
3. What service method is called?
4. What database queries run?
5. What is the response format?
Show me the relevant code in each file."

## Understand a Module
"Explain everything in src/services/. What does each service do? 
What are the key public methods? What dependencies does each have?"

## Find Patterns
"What patterns does this codebase use for:
- Error handling
- Input validation
- Authentication/authorization
- Database queries
- Testing
Show me an example of each from the actual code."

## Before Your First PR
"Review my changes and tell me:
1. Do they follow the project's coding conventions?
2. Are there any missing tests?
3. Does the error handling match the project's pattern?
4. Are there any imports from wrong packages?"
```

### 3. A Guided Exploration Session

On day one, the new developer runs this multi-step exploration:

```
Step 1: "I'm new to this project. Read the CLAUDE.md and give me 
a 2-minute summary of what this project does, what tech it uses, 
and how the code is organized."

Step 2: "Now trace the most important user flow — whatever the 
core business logic is. Show me every file involved, in order."

Step 3: "What are the 5 most important files in this project? 
For each one, explain what it does and why it matters."

Step 4: "Show me the testing setup. How do I run tests? 
Where do test files live? Walk me through one complete test file 
and explain the patterns."

Step 5: "What are the potential pitfalls in this codebase? 
Things that might surprise me or cause bugs if I'm not careful?"
```

### 4. A First Task Template

The ideal first task for a new developer using Claude Code:

```
"I need to add a 'priority' field (enum: low, medium, high, critical) 
to the Task model.

This involves:
1. Update the Prisma schema (prisma/schema.prisma)
2. Create a migration (npx prisma migrate dev --name add_task_priority)
3. Update the Zod validation schema (src/schemas/task.ts)
4. Update the service to handle the new field (src/services/task-service.ts)
5. Update the route handler (src/routes/tasks.ts)
6. Add tests for the new field (tests/routes/tasks.test.ts)
7. Update API documentation (docs/api/tasks.md)

Guide me through each step. Let me review each change before 
moving to the next step."
```

This single task teaches the new developer every layer of the stack with Claude as their guide.

## Setting Up Permissions for New Developers

New developers should start with tighter permissions. Create a conservative settings profile:

```json
{
  "permissions": {
    "allow": [
      "Read(**)",
      "Glob(**)",
      "Grep(**)",
      "Write(src/**)",
      "Write(tests/**)",
      "Bash(npm test*)",
      "Bash(npm run dev*)",
      "Bash(npm run lint*)",
      "Bash(npx prisma generate)",
      "Bash(git checkout -b*)",
      "Bash(git add*)",
      "Bash(git commit*)"
    ],
    "deny": [
      "Write(.env*)",
      "Write(prisma/migrations/**)",
      "Write(infrastructure/**)",
      "Write(.github/**)",
      "Bash(git push*)",
      "Bash(npm publish*)",
      "Bash(npx prisma migrate*)",
      "Bash(rm -rf*)"
    ]
  }
}
```

Note the extra restrictions: no pushing (they should make PRs that get reviewed), no migration editing (dangerous for production), no infrastructure changes. These guard rails relax as the developer gains experience.

## The 5-Day Onboarding Plan

```
Day 1: Setup + Explore
  Morning: Dev environment setup (with Claude helping)
  Afternoon: Guided exploration session (5 prompts above)
  
Day 2: First Task
  Full day: The "add a field" task with Claude guiding each step
  Submit first PR
  
Day 3: Deeper Exploration
  Morning: Explore the area they'll own (with Claude)
  Afternoon: Pick a "good first issue" and solve it
  
Day 4: Solo Work
  Work on a small feature independently
  Use Claude for questions instead of asking teammates
  
Day 5: Review and Adjust
  Review their Claude Code usage (what prompts worked, what didn't)
  Adjust CLAUDE.md based on questions they had
  Add any missing patterns to the onboarding section
```

## Watch Out: Over-Reliance

New developers who rely too heavily on Claude Code may not build deep understanding. Balance Claude-guided exploration with independent code reading. A good rule: use Claude to find the right files and understand the flow, then read the actual code carefully yourself.

Encourage new developers to ask Claude "why" questions, not just "how" questions:

```
❌ "Write the code to add a priority field"
✅ "Why does this project use the service pattern instead of 
    putting business logic in route handlers?"
```

## Key Takeaway

Claude Code transforms onboarding from a passive documentation-reading exercise into an interactive codebase exploration. The onboarding kit has four components: a CLAUDE.md with onboarding section (setup steps, exploration prompts, common gotchas), a starter prompt library (ready-made questions for codebase exploration), a guided exploration session (5-step deep dive), and a first task template (touches every layer with Claude guiding each step). This approach reduces time-to-first-PR from 5 days to 1.5 days and cuts team chat questions by 67%. The key: use Claude for "where" and "how," but encourage new developers to ask "why" to build deep understanding.
