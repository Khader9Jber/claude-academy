## Five Patterns

Every Claude Code pair programming session falls into one of five patterns. Each has its place. Using the wrong pattern wastes time or produces code nobody understands.

## Pattern 1: Navigator-Driver

**You decide. Claude implements.**

This is the workhorse pattern for experienced developers working on features with clear requirements.

### How It Works

```
You (Navigator): "We need to add rate limiting to the auth endpoints.
Use express-rate-limit. 100 requests per 15-minute window per IP.
Apply it only to POST /auth/login and POST /auth/register.
Return 429 with a JSON error response including retryAfter."

Claude (Driver): [reads existing middleware, writes the rate limiter,
applies it to the correct routes, adds error response formatting]

You: "Good, but the error response should match our AppError format.
Read src/middleware/error.ts for the pattern."

Claude: [reads the error handler, updates the rate limit response
to match the existing error format]

You: "Now add a test for the rate limit behavior."

Claude: [writes tests that verify 429 after 100 requests]
```

### When to Use

- Feature implementation with clear requirements
- Tasks where you know the architecture but want fast implementation
- Refactoring where you define the target and Claude makes the changes

### The Key Rule

Review every change before asking for the next one. If you skip review, you lose the "navigator" advantage — you are no longer steering, just accepting.

### Pro tip

State your intent before each step:

```
❌ "Add rate limiting"
✅ "Add rate limiting to auth endpoints. The goal is to prevent 
    brute force login attempts. We want per-IP limiting, not per-user."
```

The intent ("prevent brute force") helps Claude make better implementation decisions (per-IP, not per-user).

## Pattern 2: Rubber Duck

**You explain. Claude listens (and occasionally helps).**

The most underrated pattern. Use it when you are stuck.

### How It Works

```
You: "I'm trying to figure out why the order service sometimes 
creates duplicate orders. Here's what I know:

- It happens under high load (10+ orders per second)
- The duplicate orders have different IDs but identical content
- Both orders are charged to the customer
- The database has a unique constraint on (userId, productId, createdAt)
  but duplicates still get through

My theory is that two requests arrive within the same millisecond, 
so createdAt is identical and the unique constraint should catch it...
wait. The constraint is on the timestamp, which is millisecond precision.
Two requests in the same millisecond would violate the constraint and 
one would fail.

But we ARE getting duplicates. So the timestamps must be different.
That means there is a window where two requests pass the 'check if 
order exists' query before either writes.

This is a TOCTOU race condition. The fix is to either use a database 
advisory lock or a unique idempotency key."

Claude: "Your analysis is correct — this is a classic 
time-of-check-time-of-use race condition. For an e-commerce 
system, an idempotency key is the standard solution. Here's 
how to implement it..."
```

Notice: you solved the problem while explaining it. Claude just confirmed and offered implementation help.

### When to Use

- Debugging complex issues where the cause is unclear
- Design decisions where you need to think through tradeoffs
- Architectural questions where articulating the problem reveals the answer

### The Process

1. Explain the problem as if teaching it to someone
2. State what you know and what you have tried
3. Form a hypothesis out loud
4. Let Claude validate or challenge your hypothesis
5. Implement the solution together

## Pattern 3: Ping-Pong

**Alternating turns. You write some code, Claude writes some code.**

This pattern works well for tasks that require both human judgment and Claude's implementation speed.

### How It Works

```
You: [write the database schema for a new feature]
Claude: [generate the service layer based on the schema]
You: [review the service, adjust the business logic]
Claude: [generate the route handlers based on the service]
You: [review the routes, add auth requirements]
Claude: [generate the test suite]
You: [review tests, add edge cases Claude missed]
```

### When to Use

- Features where some parts need human creativity (schema design, business logic) and others need implementation speed (route handlers, tests)
- Learning new technologies — you write the tricky parts, Claude handles the boilerplate
- Refactoring where you make the structural decisions and Claude propagates changes

### The Rhythm

1. You handle the decisions (what to build, how to structure it)
2. Claude handles the implementation (writing the code that follows the structure)
3. You refine (catch edge cases Claude missed, adjust business logic)
4. Claude propagates (apply your refinements consistently)

## Pattern 4: Full Delegation

**Claude builds. You review.**

The fastest pattern, but use it carefully.

### How It Works

```
You: "Add complete CRUD operations for a 'categories' resource.
Follow the same patterns as the 'products' resource.
Include: Prisma schema, service, routes, validation, tests, 
and API documentation."

Claude: [reads the products resource, replicates the pattern
for categories, creates all files]

You: [review every file, test the endpoints, verify the behavior]
```

### When to Use

- Well-established patterns being replicated (CRUD for a new resource)
- Boilerplate generation (test suites for existing code)
- Tasks where the pattern is more important than the specifics

### The Danger

Full Delegation is seductive. "Claude can do it faster" leads to code that nobody understands. The developer cannot explain why certain error handling exists, what edge cases are covered, or how the code interacts with other systems.

### The Rule

After full delegation, you MUST read every line as if you wrote it. Ask yourself: "Can I explain this to a teammate? Can I debug this at 2 AM when it breaks?"

If the answer is no, switch to Navigator-Driver and redo the parts you do not understand.

## Pattern 5: Writer/Reviewer

**One Claude writes. A fresh Claude reviews.**

This is the team-scale evolution of Full Delegation — the same safety net, just automated. The writer session focuses on producing the change. A second session with a clean context loads the diff and critiques it as if a senior teammate just walked up to review.

### How It Works

```
Session A (Writer):
  "Implement the password reset flow per the spec in docs/auth-reset.md.
  Add the routes, service, tests, and wire it into the existing mailer."

[Session A produces the diff]

Session B (Reviewer — new session, fresh context):
  "Review the changes in git diff main...HEAD against docs/auth-reset.md.
  Flag: missing auth checks, race conditions, test gaps, deviations from
  CLAUDE.md conventions. Be skeptical — the writer was confident."
```

Session B has no sunk cost. It never watched the writer convince itself the code was fine, so it is far more likely to catch the "I forgot the rate limit" class of bug.

### Why Fresh Context Matters

Inside one session, Claude accumulates assumptions from its own earlier output. If it decided on turn 3 that a field is validated upstream, it will trust that assumption on turn 30 even when it is wrong. A reviewer session reads the diff without that history — it re-derives every assumption from the code.

### When to Use

- Any change going to production where a human reviewer is not available right now
- High-stakes code: auth, payments, migrations, security
- As the automated gate in a CI-driven review (see the code-review-standards lesson)
- After Full Delegation — always. Never merge fully delegated code without a fresh-context review

### The Rule

The reviewer session must not see the writer session's transcript. If both sessions share context, you get one Claude agreeing with itself. The whole value is the independent look.

## Choosing the Right Pattern

| Situation | Pattern | Why |
|---|---|---|
| Feature with clear requirements | Navigator-Driver | You direct, Claude implements |
| Stuck on a design decision | Rubber Duck | Explaining reveals the answer |
| Mixed creativity + implementation | Ping-Pong | Each does what they are best at |
| Repeating an established pattern | Full Delegation | Speed matters, pattern is proven |
| High-stakes or fully delegated change | Writer/Reviewer | Fresh context catches what the writer missed |
| Junior developer learning | Navigator (junior)-Driver (Claude) | Builds junior's decision-making |
| Debugging production issue | Rubber Duck → Navigator | Analyze first, fix second |
| Large refactoring | Navigator-Driver | You steer the refactor, Claude makes changes |
| Writing tests for existing code | Full Delegation + Writer/Reviewer | Pattern is well-defined, review catches gaps |

## Anti-Patterns to Avoid

### Anti-Pattern 1: The Clipboard Developer

```
❌ Copy Claude's code without reading it
❌ Ask Claude to "fix it" when something breaks without understanding why
❌ Treat Claude's output as infallible
```

### Anti-Pattern 2: Over-Directing

```
❌ "Write a function called processOrder that takes order as an 
    OrderInput type and returns a Promise<Order> using async/await 
    with a try-catch that catches AppError..."

This is just typing code through Claude. Write it yourself.
```

### Anti-Pattern 3: No Context Switching

Using Navigator-Driver for everything — including problems where you are stuck and should be rubber ducking, or repetitive tasks where full delegation is appropriate.

### Anti-Pattern 4: Never Reviewing

```
❌ /review → "Looks good" → merge

If you are not going to read Claude's review carefully and think 
about whether you agree, skip the automated review. A rubber-stamped 
review is worse than no review — it creates false confidence.
```

## Key Takeaway

Effective Claude pair programming requires choosing the right pattern for the situation. Navigator-Driver gives you strategic control while Claude handles implementation speed. Rubber Duck uses Claude as a sounding board that helps you solve problems through articulation. Ping-Pong alternates between human creativity and Claude implementation. Full Delegation maximizes speed but requires thorough post-review. Writer/Reviewer adds a fresh-context second Claude that catches what the writer could not see — use it as the default gate on any delegated change. The anti-pattern to watch for: treating Claude as a code vending machine instead of a thinking partner. The best results come from staying engaged — reviewing every change, understanding every decision, and switching patterns when the current one is not working.
